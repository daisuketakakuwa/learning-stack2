# 最初に「更新ロックは開発/運用の2観点で意識すべきこと」である
更新(INSERT/UPDATE/DELETE)は

- アプリから実行するSQL
- 運用対応で人が実行するSQL

の２種類あるので、両方に通ずる話。

# 本題↓ RDBでロック発生するSQL操作について

前提：
- ロック種別：
  - X = 排他ロック = eXclusive
  - S = 共有ロック = Shared
- 対象行に対するロックのみ記載
- 挿入行へのXロックは明示しない（常に発生するため）

---

# 🔹 INSERT

| 実行操作 | 対象行に加えてロックが発生するテーブル（種別） |
|-----------|----------------------------------------------|
| 単一テーブルにINSERT | なし |
| 親テーブルにINSERT | なし |
| 子テーブルにINSERT | 親（参照行：S） |

# 🔹 UPDATE

| 実行操作 | 対象行に加えてロックが発生するテーブル（種別） |
|-----------|----------------------------------------------|
| 単一テーブルUPDATE | なし |
| 親テーブルUPDATE | なし |
| 子テーブルUPDATE（FK列変更なし） | なし |
| 子テーブルUPDATE（FK列変更あり） | 親（旧参照行：S・新参照行：S） |

# 🔹 DELETE

| 実行操作 | 参照制約（子テーブル側） | 対象行に加えてロックが発生するテーブル（種別） |
|-----------|--------------------------|----------------------------------------------|
| 単一テーブルDELETE | - | なし |
| 親DELETE | RESTRICT | 子（参照確認行：S） |
| 親DELETE | CASCADE | 子（削除対象行：X） |
| 親DELETE | SET NULL | 子（更新対象行：X） |
| 子テーブルDELETE | - | なし |


# 🔥 操作別に見る「ロックが危険になる」パターン

| 操作 | 止める可能性がある処理 | ロック観点でこの操作が危険になる条件 |
|------|------------------------|----------------------|
| 子INSERT（job_permission） | 親UPDATE / 親DELETE | ・大量INSERTを1トランザクションで実行する（長い）<br>・APIなどから高頻度で登録される（多い）<br>・親が画面などから頻繁に更新される（多い） |
| 親UPDATE（employees） | 子INSERT | ・画面から頻繁に実行される（多い）<br>・TX内で重い処理を含む（長い）<br>・子がリアルタイムでINSERTされる（多い） |
| 親DELETE（employees） | 子INSERT | ・棚卸し等でまとめて削除する（長い）<br>・論理削除を一括更新する（長い）<br>・子が随時INSERTされる（多い） |
