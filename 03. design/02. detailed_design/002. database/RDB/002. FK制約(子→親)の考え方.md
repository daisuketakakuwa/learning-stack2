## 🔥 まず最初に「ロックさせたくないからあえてFK制約を設定しない」という考え方がある

FK制約は整合性を担保する強力な仕組みだが、  
**ロック競合の観点では影響範囲を広げる要因にもなる。**

### なぜか？

子テーブルに INSERT する際、DBは必ず以下を行う。

- 親レコードの存在確認
- 親レコードへのロック取得（共有ロック等）

そのため、次のような構造では親子間でロック競合が発生する可能性がある。

- 親が高頻度UPDATEされる
- 子が大量INSERTされる（バッチ）
- 親DELETEをまとめて実行する

---

### あえてFKを張らない設計が検討対象になるケース

- 親：高頻度更新されるトランザクションテーブル
- 子：大量バッチINSERTされるログ・履歴テーブル
- 親：論理削除のみ（物理削除しない）
- アプリケーション側で整合性チェックを担保できる

このような場合、

> データ整合性の強制よりも、可用性・性能を優先する

という設計判断として  
**FK制約をあえて設定しない** という選択肢がある。

---

### ⚠ 注意点

FKを張らない場合は、次のリスクを許容することになる。

- 孤児レコード発生リスク
- アプリバグによる不整合
- データ修復コスト増大

したがって、

> 「ロック影響」と「整合性担保」のトレードオフを明示的に判断すること。

## 子テーブル側の `client_id` に参照制約を設定

### RESTRICT

親テーブルに紐づけて履歴・証跡のために残しておきたいとき。

```sql
CONSTRAINT clients_fk
  FOREIGN KEY (client_id)
  REFERENCES public.clients (id)
  ON DELETE RESTRICT
  ON UPDATE RESTRICT  -- ID変更は基本「ない前提」でOK
```

---

### CASCADE

親の動きに合わせる  
（親が消えるなら子も消える、親が変わるなら子も変わる）

```sql
CONSTRAINT clients_fk
  FOREIGN KEY (client_id)
  REFERENCES public.clients (id)
  ON DELETE CASCADE
  ON UPDATE CASCADE  -- ID変更は基本「ない前提」でOK
```

---

### SET NULL

親が変わる / 消える → 子をNULLにする。  
関係は切れるが、分析用途などでデータ自体は残したい場合。

```sql
CONSTRAINT clients_fk
  FOREIGN KEY (client_id)
  REFERENCES public.clients (id)
  ON DELETE SET NULL
  ON UPDATE SET NULL  -- ID変更は基本「ない前提」でOK
```
