# アプリケーション内部（Java / JavaScript）での文字数カウント

[002. 文字コード.md](https://github.com/daisuketakakuwa/learning-stack2/blob/main/02.%20principles/general/002.%20%E6%96%87%E5%AD%97%E3%82%B3%E3%83%BC%E3%83%89.md)前提でこのページをまとめています。

## Javaの場合

- `String` は内部的にUTF-16ベース
- `length()` は **UTF-16コードユニット数**

例：

```java
"👍".length()  // 2
```

理由：
👍 (U+1F44D) は 16bitで収まらない → サロゲートペア

### コードポイント数を取得するには

```java
"👍".codePointCount(0, str.length())
```

👉 これが「Unicodeコードポイント数」

---

## JavaScriptの場合

JSも内部はUTF-16。

```javascript
"👍".length  // 2
```

コードポイント数にするなら：

```javascript
[..."👍"].length  // 1
```

---

# 🔥 重要な違い（length関数はコードユニットで文字数カウントする)

| 概念 | 👍 の扱い |
|------|-----------|
| UTF-16コードユニット | 2 |
| Unicodeコードポイント | 1 |
| グラフェム（見た目） | 1 |

---

# 4️⃣ DB側の文字数制限

## VARCHAR(100) は「コードポイントベースでの100文字」として扱う👉

例：

```sql
VARCHAR(100)
```

多くのRDB（MySQL / PostgreSQL）では：

> 🔥 「100文字」＝**コードポイントベース**の文字数

※バイト数ではない（UTF-8でもOK）

---

## ただし内部では？

保存時の流れ：

```
Java String
   ↓
JDBCでUTF-8にエンコード
   ↓
バイト列としてDBへ送信
   ↓
DBが文字として解釈
   ↓
VARCHAR制限チェック
```

---

# 5️⃣ バリデーションのベストプラクティス

## 🎯 原則

> DBが最終的な真実

---

## 🔥推奨設計

1. DBに `VARCHAR(100)` を設定
2. BEでコードポイント数チェック
3. FEでもコードポイント数チェック
4. 正規化（NFC）して保存

## 大半は、コードポイント数＝コードユニット数になる -> 絵文字(🔥)だとカウントが異なってしまう。
アプリ(FE/BE)で丁寧に文字数チェックするなら「コードポイント数」でカウントすべき。

---

## なぜコードポイント基準？

理由：

- DBの文字数概念に近い
- 言語間で再現しやすい
- 絵文字事故を防げる

---

# 6️⃣ 全レイヤーの対応関係

| レイヤー | 扱う単位 |
|----------|----------|
| CPU | ビット |
| メモリ | バイト |
| UTF-8 | コードポイント→バイト変換 |
| Java/JS内部 | UTF-16コードユニット |
| DB | 文字数（コードポイント相当） |

---

# 🔥 重要な設計思想

文字列制限を考えるとき：

> 「何を1文字と数えるのか」を明確にする

- バイト数？
- コードユニット？
- コードポイント？
- 見た目の文字？

実務では：

> ✅ Unicodeコードポイント基準が最も安全

---

# まとめ

- PCはバイトしか扱わない
- Unicodeは「文字の番号」
- UTF-8は番号をバイトに変換するルール
- Java/JSは内部UTF-16
- DBは文字数制限を持つ
- バリデーションは「同じ概念」で揃えることが重要
