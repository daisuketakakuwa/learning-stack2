# 技術的論点

前提：  
検索エンジン（ElasticSearch）を利用して検索処理を行う。

---

## ページングにおける技術的問題

ここで扱うページングにおける２つの問題。

1. **検索エンジンにない情報で間引かれる場合に総件数をどう表示するか**
2. **データ洗い替えによるページング位置のズレ問題(行の位置がズレる)**

---

# 1. 総件数の整合性問題

```
ページング
　│
　└─検索エンジンにない情報（例：権限情報）で間引く場合
        └─検索エンジンの total と表示件数が一致しない
```

## 例

- 検索エンジンのヒット数：100件  
- アプリ側で権限チェック：20件除外  
- 実際の表示：80件  

このとき：

- 「100件中80件表示」とするのか  
- 「80件」と表示するのか  

という仕様判断が必要になる。

---

## 解決策

### 解決策1：総件数を表示しない

- 無限スクロール型にする⭐️
- 件数非表示の仕様にする

メリット：
- 総件数の整合性問題が表面化しない
- 実装がシンプル

デメリット：
- 根本解決ではない
- 業務系システムではUX的に不向きな場合が多い

---

### 解決策2：検索エンジンに権限情報を持たせる

- `visibleRoles` などをインデックスに含める
- 検索時に filter 条件として利用する

例：

```json
{
  "term": { "visibleRoles": "USER" }
}
```

メリット：
- total と表示件数が一致する
- ページングが安定する
- 設計として最も整合性が高い

デメリット：
- 権限更新時の再インデックス設計が必要
- データ同期戦略が必要

---

# 2. データ洗い替えによるページング位置のズレ問題

```
OFFSETベースページング
　└─データ追加/削除/更新により位置がズレる
```

## 例

```sql
SELECT ... LIMIT 20 OFFSET 40;
```

- 前ページ取得後にデータが追加
- 次ページ取得時に重複や取りこぼしが発生

原因：
OFFSETは「位置」で取得しているため。

---

## 解決策

### 解決策1：ズレを許容する

- **✅実装が容易**
- **✅多くのWebアプリで採用されている**
- SNS系では一般的

メリット：
- 実装コストが低い

デメリット：
- 厳密性は保証されない

---

### 解決策2：カーソルベースページネーション

- `search_after` などを利用
- 「位置」ではなく「値」でページング

例：

```sql
WHERE (updated_at, id) > (?, ?)
ORDER BY updated_at, id
LIMIT 20;
```

メリット：
- 重複・取りこぼしが大幅に減る
- パフォーマンスが安定
- 大規模データで有利

デメリット：
- 実装がやや複雑
- ソートキー設計が重要（ユニークキーを必ず含める）

---

# 結論

- 業務システムでは「権限込み検索 + カーソル」が安定解
- SNS系では「ズレ許容 + 無限スクロール」が一般的
- 完全一貫性が必要な場合のみ Snapshot 検索を検討する
