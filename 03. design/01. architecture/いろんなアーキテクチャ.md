# いろんなレイヤーに「アーキテクチャ」が存在する

|レイヤー|ラフな説明|詳細|
|----|----|----|
|System|外から見た全体構成|**アプリとインフラを含めた**“システム全体”の構成|
|Application|アプリ間の構成|**デプロイ単位**のアプリ同士の関係と責務の整理|
|Software|アプリ内部の構成|1つのアプリ**内部のコード構造設計**|


```
System Architecture🔴
  ├─ Application Architecture🔵
  │   ├─ Frontend
  │   │   └─ Frontend Software Architecture🟢
  │   ├─ Backend
  │   │   └─ Backend Software Architecture🟢
  │   └─ ⭐️Database (show data ownership)
  └─ Infrastructure Architecture🔵
      ├─ Frontend runtime
      ├─ Backend runtime
      ├─ ⭐️Database (infra layer)
      └─ CDN / Load Balancer / etc.
```

1. システム = 「アプリ」＋「インフラ」を含めたアーキテクチャ
2. アプリ   = 「FEアプリ」＋「BEアプリ」＋「DB(アプリレイヤー)」を含めたアーキテクチャ
3. ソフト   = FE、BEそれぞれに閉じたソフトに関するアーキテクチャ

## システムアーキテクチャ
```
┌─────────────────────────────┐
│ User / Client               │ ← ブラウザ・モバイル
├─────────────────────────────┤
│ CDN / Load Balancer         │ ← リクエスト分散、キャッシュ
├─────────────────────────────┤
│ Web / Application Servers   │ ← BEのSWAが稼働するサーバ群
├─────────────────────────────┤
│ Database Cluster / Cache    │ ← 永続化・高速アクセス
├─────────────────────────────┤
│ Message Queue / Event Bus   │ ← 非同期処理・連携
├─────────────────────────────┤
│ External Systems / APIs     │ ← 外部サービス、他システム
└─────────────────────────────┘
```


## アプリケーションアーキテクチャ(“What are the deployable apps?”)

　Application = **deploy as a unit**（**開発の中で定期的にリリースが発生する単位**）

So they are apps -> FE app, BE app, worker, batch<br>
So they are NOT apps -> Database, cache<br>

DB is not an application, but we need it to clarify 「**Who owns data**」.

### Application Architecture defines:
- Deployment units -> ✅FE app, BE app
- Interactions -> ✅FE --HTTP/GraphQL--> BE ---DB/External API
- Who owns what data? -> ✅RDB

```
                         ┌────────────────────────────┐
                         │        External User        │
                         └──────────────┬─────────────┘
                                        │
                                        ▼
                        ┌─────────────────────────────┐
                        │        Frontend App         │
                        │  - SPA / Mobile / Web       │
                        │  - Release unit             │
                        └──────────────┬──────────────┘
                                       │ HTTP / GraphQL
                                       ▼
        ┌─────────────────────────────────────────────────────┐
        │                 Backend Application                 │
        │  - Deployable service                               │
        │  - Owns business logic                              │
        │  - Exposes API                                      │
        │  - Versioned & released independently               │
        └──────────────┬───────────────────────┬──────────────┘
                       │                       │

           ┌──────────────────┐     ┌────────────────────────┐

```

## ソフトウェアアーキテクチャ（“How is each app internally structured?”)
How the application is structured internally at the code level. <br>
✅Think about **Internal structure of the application**.（アプリのソースコード構成について）<br>
✅Scope = **Inside one service**.（1サービス内部に閉じた話）<br>
✅Level = **Implementation-Level**.（実装レベル）<br>

### FEのSWA
```
Frontend Application (Deployable Unit) ※Exampl as layered model
│
├── Presentation Layer
│   ├── Pages / Screens
│   └── UI Components
│
├── Application Layer
│   ├── State Management
│   ├── Hooks / UseCases
│   └── Routing
│
├── Domain Layer (optional but recommended)
│   ├── Business Rules
│   ├── Domain Models
│   └── Validation Logic
│
└── Infrastructure Layer
    ├── API Client
    ├── Storage (LocalStorage, Cookie)
    └── External SDK Wrappers
```
**Frontend SWA defines:**
- Layering Strategy（Atomic design? Layerd-based? Feature-based?
- State Management（Global vs Local, Redux vs Contet）
- Folder Structure Rules
- Component Strategy（use UI library vs create from scratch?


### BEのSWA
```
Backend Application (Deployable Unit)
│
├── Presentation Layer
│   ├── Controllers
│   ├── DTOs
│   └── Request/Response Mapping
│
├── Application Layer
│   ├── UseCases
│   ├── Transaction Control
│   └── Application Services
│
├── Domain Layer
│   ├── Entities
│   ├── Value Objects
│   ├── Domain Services
│   └── Business Rules
│
└── Infrastructure Layer
    ├── Repository Implementations
    ├── ORM
    ├── External API Clients
    └── Messaging Adapters
```

**Backend SWA defines:**
- Layer Model（Traditional 3-layer? Clean Architecture?
- Dependency（Can controllers access repositories directly?
- Domain Modeling（Rich domain model? Anemic model?
- Error Handling（Global handler?
- Cross-Cutting concerns（Logging, Security filters, Middleware, Aspect）

## 技術スタック(Tech stack) と SWA　がお互いに影響しあう話
```
Spring Boot → Controller / Service / Repository pattern
Next.js → File-based routing
```

👉Software Architecture and Technical Stack are independent at the conceptual level,（**概念としてお互いに独立**）<br>
👉but mutually influence each other in practical design decisions.(**実際に決める時は影響/依存し合う構造になる**)<br>
