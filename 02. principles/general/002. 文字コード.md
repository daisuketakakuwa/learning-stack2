# 💻 PCと文字コードの仕組み整理（アプリ・DB視点まで含めた全体像）

---

# 1️⃣ 大前提：PC（CPU / メモリ）はバイト列しか扱わない

コンピュータが物理的に扱えるのは最終的に：

```
01001000 01101001 ...
```

のような **ビット列（＝バイト列）** だけです。

- CPUはビット列を処理する
- メモリにはバイト列が格納される
- ファイルもバイト列で保存される

👉 「文字」は人間側の抽象概念

---

# 2️⃣ 文字が表示されるまでの流れ

✅Unicodeは「文字 --- Unicodeコードポイント(**番号**) の紐付け」<br>
✅UTF-8は「Unicodeコードポイント(**番号**) <---変換---> 16進数の**バイト列**」<br>
→ PC(CPU/メモリ)は文字列(Unicodeコードポイント)を扱えない。UTF-8でエンコードされたバイト列を扱える。

```
[人間の入力]
      ↓
[アプリ内部：Unicodeとして保持]
      ↓
[エンコード（例：UTF-8）]
      ↓
[バイト列]
      ↓
[保存 / 転送]
      ↓
[デコード]
      ↓
[画面表示]
```

重要：

> 🔥 バイト列そのものには「UTF-8」という情報は含まれない  
> 🔥 読む側がどの文字コードで解釈するかが重要

---

# 📂 ファイルの場合

## 保存時

エディタは：

1. 文字をUnicodeとして保持
2. 指定した文字コード（UTF-8など）でエンコード
3. バイト列を書き込む

例：

| 文字 | UTF-8 | UTF-16 |
|------|--------|--------|
| あ | E3 81 82 | 42 30 |

---

## 文字化けの原因

- UTF-8で保存
- UTF-16として読み込み

👉 バイト列を違うルールで解釈  
👉 意味不明な文字になる

> 🔥 文字化け = エンコードとデコードの不一致

---

# 🖥 ターミナルの場合

ターミナルにも文字コード設定がある。

## 入力

```
キーボード → Unicode → UTF-8 → バイト列 → プロセス
```

## 出力

```
プロセス出力（バイト列） → UTF-8でデコード → 表示
```

Linux/macOSでは通常：

```
LANG=ja_JP.UTF-8
```

が設定されている。

# SJIS と Unicode / UTF-8 の関係

---

# 1️⃣ まず全体像

文字コードの話は「2レイヤー」に分けると分かりやすい。

## 🔹 レイヤー1：文字集合（Character Set）
どの文字を扱えるか？という定義

例：
- ASCII
- Shift_JIS
- Unicode

---

## 🔹 レイヤー2：符号化方式（Encoding）
文字をどうやってバイト列に変換するか？

例：
- UTF-8
- UTF-16
- UTF-32

---

# 2️⃣ SJISとは何か？

正式名称：Shift_JIS

これは：

> 🔥 「文字集合」と「符号化方式」がほぼ一体化したもの

特徴：

- 日本語向け
- 可変長（1バイト or 2バイト）
- Windowsで長年使われた
- Unicodeとは別世界

---

# 3️⃣ Unicodeとは何か？

Unicodeは：

> 🔥 世界中の文字に番号（コードポイント）を振った規格

例：

| 文字 | コードポイント |
|------|----------------|
| あ | U+3042 |
| 漢 | U+6F22 |
| 👍 | U+1F44D |

Unicode自体は「番号の規格」。

まだバイトではない。

---

# 4️⃣ UTF-8とは何か？

UTF-8は：

> Unicodeのコードポイントをバイト列に変換するルール

例：

| 文字 | Unicode | UTF-8 |
|------|----------|--------|
| あ | U+3042 | E3 81 82 |
| 👍 | U+1F44D | F0 9F 91 8D |

---

# 5️⃣ SJISとUnicodeの決定的違い

| 観点 | SJIS | Unicode |
|------|------|----------|
| 対象 | 主に日本語 | 世界中の文字 |
| 絵文字 | 基本なし | ある |
| 設計思想 | ローカル | グローバル |
| 現在 | レガシー | 標準 |

---

# 6️⃣ なぜSJISは問題になるのか？

## ① 文字が足りない

UnicodeにはあるがSJISにはない文字がある。

例：
- 絵文字
- 一部の漢字
- 外国語文字

---

## ② 同じ文字でも番号が違う

例えば：

「あ」

- SJIS → 0x82A0
- UTF-8 → E3 81 82

👉 完全に別物。

---

## ③ 文字化けが起きやすい

UTF-8で保存したファイルをSJISで開くと？

→ 解釈ルールが違う  
→ 文字化け

---

# 7️⃣ 現代の標準構造

現在のWeb世界は：

```
Unicode
   ↓
UTF-8
   ↓
インターネット
```

ほぼUTF-8一択。

HTTPヘッダにも：

```
Content-Type: text/html; charset=UTF-8
```

と明示する。

---

# 8️⃣ SJISとUnicodeはどう共存する？

変換が必要。

```
SJISバイト列
    ↓ デコード
Unicodeコードポイント
    ↓ エンコード
UTF-8バイト列
```

この変換をアイコンバータ（iconvなど）が行う。

---

# 9️⃣ Java/JavaScriptとの関係

JavaやJSの内部はUnicodeベース。

つまり：

- SJISファイルを読むとき
- SJIS → Unicodeへ変換
- 内部ではUnicodeとして扱う

---

# 🔟 DBとの関係

昔のDB：

- 文字コードがSJIS

現代のDB：

- UTF-8（utf8mb4など）

もしアプリがUTF-8で、
DBがSJISだと？

👉 変換ミスで文字化け・保存失敗が起きる

---

# 🔥 まとめ

- SJISは日本語特化の古い文字コード
- Unicodeは世界共通の文字番号規格
- UTF-8はUnicodeをバイトに変換する方法
- 現代のWeb標準はUTF-8
- SJISはレガシー環境でのみ登場

---

# 🧠 一言でいうと

> SJISは「昔の日本ローカル仕様」  
> Unicode/UTF-8は「現代の世界共通仕様」
