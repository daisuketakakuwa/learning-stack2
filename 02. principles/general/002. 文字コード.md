# 💻 PCと文字コードの仕組み整理（アプリ・DB視点まで含めた全体像）

---

# 1️⃣ 大前提：PC（CPU / メモリ）はバイト列しか扱わない

コンピュータが物理的に扱えるのは最終的に：

```
01001000 01101001 ...
```

のような **ビット列（＝バイト列）** だけです。

- CPUはビット列を処理する
- メモリにはバイト列が格納される
- ファイルもバイト列で保存される

👉 「文字」は人間側の抽象概念

---

# 2️⃣ 文字が表示されるまでの流れ

✅Unicodeは「文字 --- Unicodeコードポイント(**番号**) の紐付け」<br>
✅UTF-8は「Unicodeコードポイント(**番号**) <---変換---> 16進数の**バイト列**」<br>
→ PC(CPU/メモリ)は文字列(Unicodeコードポイント)を扱えない。UTF-8でエンコードされたバイト列を扱える。

```
[人間の入力]
      ↓
[アプリ内部：Unicodeとして保持]
      ↓
[エンコード（例：UTF-8）]
      ↓
[バイト列]
      ↓
[保存 / 転送]
      ↓
[デコード]
      ↓
[画面表示]
```

重要：

> 🔥 バイト列そのものには「UTF-8」という情報は含まれない  
> 🔥 読む側がどの文字コードで解釈するかが重要

---

# 📂 ファイルの場合

## 保存時

エディタは：

1. 文字をUnicodeとして保持
2. 指定した文字コード（UTF-8など）でエンコード
3. バイト列を書き込む

例：

| 文字 | UTF-8 | UTF-16 |
|------|--------|--------|
| あ | E3 81 82 | 42 30 |

---

## 文字化けの原因

- UTF-8で保存
- UTF-16として読み込み

👉 バイト列を違うルールで解釈  
👉 意味不明な文字になる

> 🔥 文字化け = エンコードとデコードの不一致

---

# 🖥 ターミナルの場合

ターミナルにも文字コード設定がある。

## 入力

```
キーボード → Unicode → UTF-8 → バイト列 → プロセス
```

## 出力

```
プロセス出力（バイト列） → UTF-8でデコード → 表示
```

Linux/macOSでは通常：

```
LANG=ja_JP.UTF-8
```

が設定されている。
