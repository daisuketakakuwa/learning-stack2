# L4レベルのアクセス制御 と L7レベルのアクセス制御

## まず結論（ざっくり比較）

| 観点 | L4（トランスポート層） | L7（アプリケーション層） |
|------|-----------------------|--------------------------|
| 見ている情報 | IP / ポート / プロトコル | HTTPヘッダ / パス / パラメータ / ボディ |
| 主な目的 | 不正通信・DoS系の遮断 | Webアプリの脆弱性攻撃の防御 |
| 代表例 | FW / IDS / IPS | WAF |
| 例 | SYNフラッド、UDPフラッド | SQLインジェクション、XSS |

---

## L7（アプリケーション層）: WAF

### WAF（Web Application Firewall）
**役割**  
- HTTPリクエストの「中身」を解析して、Webアプリへの攻撃を防ぐ  
- アプリの脆弱性を“前段で補強”するイメージ

**防げる攻撃例**
- SQLインジェクション  
- XSS  
- CSRF  
- パストラバーサル  
- 設定ファイルの直アクセス  
  - 例：`http://example.com/.env`  
- 不審な User-Agent / Bot アクセス

**代表例**
- [Cloudflare](chatgpt://generic-entity?number=0) WAF  
- [AWS](chatgpt://generic-entity?number=1) WAF  
- [Akamai](chatgpt://generic-entity?number=2) WAF  

**ポイント**
- L7は「アプリの文脈が分かる」  
- 誤検知はあるが、チューニングでかなり実用レベルまで下げられる  
- 認証前後のリクエスト制御、Bot対策なども可能

---

## L4（ネットワーク層）: IDS / IPS

### IDS（Intrusion Detection System）
**役割**  
- 不正通信を検知して通知する（止めない）  
- SOC / SIEM へのアラート連携向き

**特徴**
- ◎ 誤検知があっても通信は遮断しない → 業務影響が少ない  
- △ “検知だけ”なので防御は別途必要  

**代表例**
- [AWS](chatgpt://generic-entity?number=3) [Amazon GuardDuty](chatgpt://generic-entity?number=4)  
  - VPC Flow Logs / CloudTrail / DNS ログなどを解析するマネージドIDS的サービス

---

### IPS（Intrusion Prevention System）
**役割**  
- 不正通信を「検知＋遮断」する  
- FWより賢く、WAFより低レイヤー

**特徴**
- ◎ 攻撃をリアルタイムにブロックできる  
- △ 誤検知すると“正当通信まで遮断”されるリスクあり  
- 本番環境では慎重なルール設計が必須

**AWS構成例（疑似IPS構成）**
- GuardDuty → EventBridge → Lambda → Security Group更新  
  - GuardDutyは単体では遮断しない  
  - Lambdaなどと組み合わせて“自動遮断”することで IPS的に振る舞わせる

## ネットワーク層（L4）における代表的な攻撃例

> ※ 以下は**学習目的の説明**。実運用環境での実行はNG。

### 🔴 SYNフラッド（TCP枯渇攻撃）
- TCP 3-way handshake を悪用  
- SYNだけ大量送信 → サーバ側の接続待ちキュー枯渇  
- 結果：正規ユーザーが接続できなくなる

### 🔴 UDPフラッド
- UDPはコネクションレス  
- ひたすら大量送信 → 帯域・CPU枯渇  
- レスポンス不要なため攻撃コストが低い

### 🔴 ICMPフラッド（Ping flood）
- ICMP Echo Request を大量送信  
- ネットワーク帯域やOSリソースを消費
