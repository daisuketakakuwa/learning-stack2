# Webアプリ、ID/PW認証後のセッション管理方式

## はじめに

Webアプリの認証を考えるとき、

```text
認証方式
↓
ログイン後のセッション管理方式
```

は別物として考える必要がある。

例えば、

```text
ID/PW認証
Passkey認証
Email OTP認証
```

は「本人確認の方法」であり、

```text
ログイン後にどうやってログイン状態を維持するか
```

とは別の話である。

本記事では、

```text
ID/PW認証成功後、
どうやってログイン状態を維持するか
```

に焦点を当てる。

---

# セッション管理方式一覧

大きく分けると以下のようになる。

```text
認証成功
↓
セッション管理

├ Session認証（従来型）
│
├ JWT認証
│   ├ Authorization: Bearer🔴localStorage等にJWT格納
│   └ HttpOnly Cookie✅CookieにJWT格納
│
└ OAuth/OIDC
     └ （内部的にはJWTを利用することが多い）
```

---

# 1. Session認証（従来型）

最も古くから使われている方式。

## 仕組み

```text
ログイン成功
↓
サーバでSession生成
↓
SessionID発行
↓
Cookie保存
↓
以後SessionIDを送信
```

例

```http
Set-Cookie:
JSESSIONID=abc123
```

ブラウザ

```http
Cookie:
JSESSIONID=abc123
```

サーバ

```text
SessionID
↓
セッションストア検索
↓
ユーザー特定
```

---

## 特徴

### メリット

- 実績が豊富
- 実装がシンプル
- 強制ログアウトしやすい

### デメリット

- サーバ側でセッション管理が必要
- Redis等が必要になることがある
- スケール時に工夫が必要

---

## 状態管理

```text
Stateful
```

---

# 2. JWT認証

近年広く利用されている方式。

## 仕組み

```text
ログイン成功
↓
JWT発行
↓
クライアント保存
↓
毎回JWT送信
↓
JWT検証
```

例

```text
eyJhbGciOi...
```

JWTの中に

```json
{
  "sub": "user001",
  "role": "ADMIN",
  "exp": 1750000000
}
```

などを保持する。

---

## 特徴

### メリット

- サーバ側セッション不要
- マイクロサービス向き
- APIサーバとの相性が良い

### デメリット

- 失効管理が難しい
- JWTサイズが大きい
- 保存場所を考慮する必要がある

---

## 状態管理

```text
Stateless
```

---

# JWTの保存・送信方法

JWT認証には大きく2パターンある。

```text
JWT認証

├ Bearer認証
└ Cookie-based JWT Authentication
```

---

# 2-1. Bearer認証

JWTをクライアントで保持し、

Authorizationヘッダで送信する。

## 仕組み

```text
ログイン成功
↓
JWT発行
↓
localStorage等へ保存
↓
Authorizationヘッダ付与
```

例

```http
Authorization: Bearer eyJhbGciOi...
```

---

## メリット

- CSRFに強い
- API設計がシンプル
- モバイルアプリと相性が良い

---

## デメリット

### XSSに弱い

例えば

```js
localStorage.getItem("token")
```

で取得可能。

XSSが発生すると

```text
JWT窃取
↓
アカウント乗っ取り
```

のリスクがある。

---

# 2-2. Cookie-based JWT Authentication

現在のWebアプリで有力な選択肢。

## 仕組み

```text
ログイン成功
↓
JWT発行
↓
HttpOnly Cookie保存
↓
ブラウザ自動送信
↓
JWT検証
```

例

```http
Set-Cookie:
access_token=xxxxx;
HttpOnly;
Secure;
SameSite=Lax
```

---

## メリット

### XSS耐性

```js
document.cookie
```

で取得できない。

JWTの窃取が困難。

---

### Cookie属性を活用できる

```text
HttpOnly
Secure
SameSite
```

などを利用可能。

---

### Stateless

JWTなので

```text
サーバ側セッション不要
```

を維持できる。

---

## デメリット

### CSRF対策が必要

Cookieは自動送信されるため、

```text
CSRF Token
Originチェック
SameSite
```

などの対策が必要。

---

# OAuth/OIDCとの関係

よく混同されるが、

```text
OAuth
OIDC
```

はセッション管理方式ではない。

---

OAuth/OIDCは

```text
どのように認証・認可を行うか
```

を定義するプロトコル。

---

その結果取得したJWTを、

```text
Authorization: Bearer
```

で送ることもできるし、

```text
HttpOnly Cookie
```

へ保存することもできる。

---

# 比較表

| 項目 | Session認証 | Bearer認証 | Cookie-based JWT |
|--------|--------|--------|--------|
| 状態管理 | Stateful | Stateless | Stateless |
| サーバ側セッション | 必要 | 不要 | 不要 |
| XSS耐性 | ◎ | × | ◎ |
| CSRF耐性 | △ | ◎ | △ |
| 強制ログアウト | ◎ | △ | △ |
| マイクロサービス適性 | △ | ◎ | ◎ |
| Webアプリ適性 | ◎ | ○ | ◎ |
| モバイルアプリ適性 | △ | ◎ | △ |

---

# ブラウザ向けWebアプリ におけるセッション管理方法の主流

ブラウザ向けWebアプリでは、

```text
Cookie-based JWT Authentication

(JWT + HttpOnly Cookie)
```

が有力な選択肢となっている。

理由は、

```text
Bearer認証のXSSリスクを減らしつつ、
JWTのStateless性を維持できる
```

ためである。

特に、

```text
React
Vue
Angular
+
Spring Boot
Node.js
Go
```

といったSPA + API構成ではよく採用される。
